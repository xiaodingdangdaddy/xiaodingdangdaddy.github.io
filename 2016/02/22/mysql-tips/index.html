索引介绍

资料库和表格时MySQL数据库最基本的原件，设计良好的数据库，无论是资料的正确性还是后续的维护查询都不会发生什么问题。
利用索引index除了可以预防数据出现问题外，在表格中记录量很大情况下，可以增加查询与维护资料的效率。
对于以上的Store_Information表格执行
SELECT * FROM Store_Informatin WHERE Store_Name = ‘Boston’
如果没有索引帮助，引擎就会从头开始一边读取一边判断。如果建立Store_Name的索引，引擎就会从索引档中搜寻，找到后，再从索引档对照到表格记录。
tips:索引同样可以增加删除或者修改的效率
索引分为三种：
1主键索引(primary key),一个表格只能有一个，设定为主键索引的栏位值不可重复，也不可以储存NULL。因此适合在编码、代号、身份证号这种栏位
2唯一索引(unique index),设定为唯一索引的栏位值同样不可重复，但是可以为NULL.适合用在类似员工资料表格中，存储邮箱号，电话号码等，允许NULL值的栏位
3非唯一索引(non-unigue index),只是用来增加查询维护资料的效率，无其他限制。

1使用’CREATE INDEX’在建立表格后，可以建立唯一索引和非唯一索引。

ANY 关键词的用法

ANY关键词必须后面接一个比较操作符(即>,<,=等)。ANY关键词的意思是“对于在子查询返回的列中的任一数值，例如：
SELECT s1 FROM t1 WHERE s1 > ANY (SELECT s1 FROM t2);
IN是”＝ANY”的别名。因此，这两个语句是一样的：
SELECT s1 FROM t1 WHERE s1 = ANY (SELECT s1 FROM t2);
SELECT s1 FROM t1 WHERE s1 IN (SELECT s1 FROM t2);

关于行级锁、表级锁和页级锁

行级：InnoDB引擎.单独对一行记录加锁。由于仅对指定记录加锁，这样其他进程还是可以对同一个表中的其他记录进行操作。特点：开销大，加锁慢。会出现死锁。锁定粒度最小，发生锁冲突的概率最低，并发度也最高。适合于有大量按索引条件并法更新少量不同数据，同时又有并发查询的应用，如一些在线事务处理系统(OLTP).
页级：BDB引擎.一次锁定相邻的一组记录。由于表级锁速度快但冲突多，行级锁冲突少但速度慢。页级锁等于是个折中的做法。开销和加锁时间介于行级和表级之间。会出现死锁，锁定粒度也介于二者之间，并发度一般。
表级：MyISAM引擎.直接锁定整张表，在锁定期间，其他进程无法对该表进行写操作。开销最小，加锁最快。不会出现死锁。锁定粒度大，发生锁冲突概率最高，并发度最低。适合于以查询为主，只有少量按索引条件更新数据的应用，如web应用。

Mysql5.5.5版本开始默认引擎为InnoDB。
行级锁的优点：
1当在许多线程中访问不同的行时，只存在少量的锁定冲突。
2回滚时只有少量的更改。
3可以长时间锁定单一的行
行级锁的缺点：
1比页级和表级锁占用更多的内存。
2当在表的大部分中使用时，比页级和表级要速度慢，因为你必须获取更多的锁。
3如果在大部分数据上常使用GROUP BY等必须扫描整个表的操作时，比其他锁要明显慢很多。
InnoDB和MyISAM锁的最大不同在于：一是支持事务TRANSACTION，二是采用行级锁。
事务是由一组SQL语句组成的逻辑处理单元，有四个属性，简称ACID属性：
1原子性Atomicity.事务是一个原子操作单元，其对数据的修改，要么全部执行，要么全部不执行。
2一致性Consistent.在事务开始和完成时，数据必须保持一致状态。
3隔离性Isolation.数据库系统提供一定的隔离机制，保障事务在不受外部并发操作影响的‘独立’环境执行。
4持久性Durable.事务完成后，它对于数据的修改时永久的，即使出现系统故障也能保持。

InnoDB有两种模式的行锁：
1共享锁。允许一个事务读取一行，阻止其他事务获得相同数据集的排它锁。
2排他锁。允许获得排他锁的事务更新数据，阻止其他事务取得相同数据集的共享读锁和排他写锁。
InnoDB的行锁是通过给索引项加锁来实现的。即只有通过索引条件检索数据，才能使用行级锁，否则使用表锁.
